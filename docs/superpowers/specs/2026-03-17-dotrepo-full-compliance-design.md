# dotrepo Full Compliance — Design Spec

**Date:** 2026-03-17
**Status:** Approved

---

## Goal

Bring `ries-rs` to full dotrepo v0.1 compliance:
- Fix the rendering gap in `.repo` (`readme.sections` references undefined custom section keys)
- Make `.repo` the single source of truth for all GitHub community files
- Enable dotrepo's generation surface for `CONTRIBUTING.md`, `.github/SECURITY.md`, `CODEOWNERS`, and the PR template

---

## Background

The existing `.repo` record is structurally sound (`mode = "native"`, `status = "canonical"`, all required sections present) but has two issues:

1. **Renderer gap**: `readme.sections` references `install`, `documentation`, and `citation` — none have `[readme.custom_sections]` entries. `dotrepo validate` does not currently error on this, but `dotrepo generate` cannot render those sections without the content definitions.
2. **Generation surface unused**: All four `[compat.github]` keys are `"skip"`, so dotrepo does not own any community files. `CONTRIBUTING.md` exists but is manually maintained; `.github/SECURITY.md`, `CODEOWNERS`, and `.github/pull_request_template.md` are absent.

---

## Approach

**Option chosen: Full `.repo` as source of truth (Option 2)**

Update `.repo` completely and regenerate all four community files from it. The existing `CONTRIBUTING.md` content is preserved and incorporated into the regenerated version.

Alternatives considered and rejected:
- *Minimal (Option 1)*: hand-crafted "generated" files drift from `.repo` over time.
- *Conservative (Option 3)*: leaves `CONTRIBUTING.md` unmanaged, which defeats the point.

---

## Design

### Part 1: `.repo` changes

Four targeted changes to the existing record:

#### 1.1 `security_contact`
Change from `"unknown"` to the GitHub security advisories URL:
```toml
security_contact = "https://github.com/maxwellsantoro/ries-rs/security/advisories/new"
```

#### 1.2 `compat.github` — all four set to `"generate"`
```toml
[compat.github]
codeowners            = "generate"
security              = "generate"
contributing          = "generate"
pull_request_template = "generate"
```

#### 1.3 `docs.api`
Point to the documentation map, which serves as the API index:
```toml
api = "docs/README.md"
```

#### 1.4 `[readme.custom_sections]` — three new entries
Define content for each currently-undefined section key so `dotrepo validate` can resolve the references:

```toml
[readme.custom_sections.install]
content = "Install via `cargo install ries --locked` (CLI), `pip install ries-rs` (Python), or download a prebuilt archive from the GitHub releases page."

[readme.custom_sections.documentation]
content = "Full documentation lives in `docs/`. Start with the [documentation map](docs/README.md) for an index of all guides."

[readme.custom_sections.citation]
content = "Cite via `CITATION.cff` (canonical). A BibTeX entry and Zenodo DOI are in the README under *How to Cite*."
```

---

### Part 2: Generated community files

All four files carry a dotrepo generation banner:
```
<!-- generated by dotrepo <tool-version> | schema: dotrepo/v0.1 | source: sha256:<digest> -->
```
where `<digest>` is the sha256 of the `.repo` file after all changes are applied, and `<tool-version>` is the semver of the dotrepo CLI used to generate (e.g. `0.1.0`).

#### 2.1 `.github/SECURITY.md` (new)
Reporting policy pointing to the GitHub security advisories URL. Derived from `[owners].security_contact`. Short policy document appropriate for a solo open-source project. dotrepo writes this to `.github/SECURITY.md` (not the repo root).

#### 2.2 `CONTRIBUTING.md` (replaced)
The existing content (prerequisites, build/test commands, code style, PR process, feature flags, benchmarks) is preserved and incorporated into the new dotrepo-managed version. No information is lost.

#### 2.3 `.github/CODEOWNERS` (new)
Single ownership rule: `@maxwellsantoro` owns `*`. Derived from `[owners].maintainers`.

#### 2.4 `.github/pull_request_template.md` (new)
Standard template covering:
- Summary of changes
- Type of change checklist (bugfix / feature / docs / refactor / other)
- Test checklist (`cargo fmt --check`, `cargo clippy`, `cargo test`)
- Documentation update reminder

## Validation

After implementation, running `dotrepo validate` should produce no errors or warnings:
- `repo.name` non-empty ✓
- `record.mode = "native"` + `record.trust` present ✓
- All `docs.*` paths exist on disk ✓
- No unexplained `unknown` placeholders ✓

Running `dotrepo generate` should successfully render all three `readme.sections` custom keys (`install`, `documentation`, `citation`) now that each has a `[readme.custom_sections]` content entry.

The four generated files must each carry the dotrepo banner referencing the sha256 of the final `.repo` content.
